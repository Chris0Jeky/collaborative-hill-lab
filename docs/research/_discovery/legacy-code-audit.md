## TL;DR (most important findings)

1. **There is no single canonical implementation.** At least **four** parallel game engines coexist, each with its own payoff and TFT semantics: (a) `npdl/core/` (the "library"), (b) `3-Person_Tragic_vs_Reciprocity/npd_simulator/` (modular rewrite), (c) `3-Person_Tragic_vs_Reciprocity/final_experimentations/v1..v9` + `legacy/` + `legacy_cleaned/` (standalone scripts), and (d) `simple_models/`. **The paper/website figures were generated by the standalone `final_experimentations`/`code_for_website` scripts (family c), NOT by `npdl/` or `npd_simulator/`.** HIGH.

2. **THE DEFECT is a cooperation-*tracking* (reporting) bug, not a behavioral one.** In `npd_simulator`, `_extract_cooperation_by_round()` expects a round-indexed history schema (`round_data["agents"][id]["action"]`) that the pairwise engine never produces (its history is pair-indexed: `agent1_id/action1/...`). The extractor silently falls through to `round_data.get("cooperation_rate", 0)`, which is also absent, so **every pairwise round is recorded as 0.0 cooperation** — including round 1, where TFT always cooperates. This contaminated the committed `test_results/pairwise_tft_cooperation_all_experiments_summary.csv` (all zeros). HIGH.

3. **The three "pairwise" implementations do not agree on what pairwise means**, and the `npdl` one is scientifically *wrong* for the paper's claim: `npdl` pairwise emits **one move per round applied to all opponents** and TFT defects if **any** opponent defected — so a single AllD collapses all TFTs to defection (the *opposite* of the "Reciprocity Hill"). Only the standalone paper code and `npdl/core/true_pairwise.py` implement genuine **per-opponent** reciprocity. A committed test (`test_neighborhood_vs_pairwise.py:70`) actively **enshrines the wrong behavior**. HIGH.

4. **"TFT" is not the same strategy across conditions.** In the paper code, pairwise TFT is deterministic per-opponent reciprocity; neighbourhood/N-person "TFT" is **probabilistic pTFT** (cooperate with probability = group cooperation ratio). The headline Hill-vs-Valley contrast is therefore partly an artifact of **two different decision rules**, not purely of interaction structure. This is the single biggest scientific caveat for reuse. HIGH.

5. **The paper-generating scripts are unseeded** — no `random.seed`/`np.random.seed` anywhere in the standalone/`v9` runners. Results are only reproducible in distribution (averaged over N runs), not exactly. The `npdl/main.py` runner *is* seeded (`random.seed(run_number)`). HIGH.

## 2. PAYOFF IMPLEMENTATIONS (verbatim, with file:line)

Canonical PD constants everywhere: **R=3, S=0, T=5, P=1** (satisfies T>R>P>S and 2R>T+S).

### (a) Pairwise (2-player) payoffs
- **`npdl/core/utils.py:138-149` and again `:152-177`** — `get_pairwise_payoffs` is **defined twice** (the second shadows the first; they're numerically identical, only docstrings/error msg differ). Table: CC→(R,R); CD→(S,T); DC→(T,S); DD→(P,P). HIGH. *Duplicate-definition smell; harmless but sloppy.*
- **`npd_simulator/core/models/payoff_matrix.py:74-88`** `get_pairwise_payoff(my_action, opp_action)`: `my=C → R if opp=C else S; my=D → T if opp=C else P`. Actions here are **0=cooperate, 1=defect** (opposite polarity to the string API). HIGH.
- **Paper code:** `final_simulation.py:6` and `single_run_3tfte_standalone.py:18-21`: `PAYOFFS_2P = {(0,0):(3,3),(0,1):(0,5),(1,0):(5,0),(1,1):(1,1)}`. HIGH.

### (b) Neighbourhood / N-person (linear) payoffs — **the canonical N-person rule**
- **`npdl/core/utils.py:6-20`**
  ```python
  def linear_payoff_C(n, N, R=3, S=0):
      return S + (R - S) * (n / (N - 1)) if N > 1 else R
  def linear_payoff_D(n, N, T=5, P=1):   # :23-37
      return P + (T - P) * (n / (N - 1)) if N > 1 else P
  ```
  where `n` = number of *cooperating neighbours (others)*, `N` = group size. Also `exponential_*` (prop**exponent) and `threshold_*` variants exist (utils.py:40-135) — non-canonical, selectable via `create_payoff_matrix(payoff_type=...)`.
- **`npd_simulator/core/models/payoff_matrix.py:46-72`**: `linear_cooperator_payoff(ratio)=S+(R-S)*ratio`; `linear_defector_payoff(ratio)=P+(T-P)*ratio`, with `ratio = num_others_cooperating/(N-1)` computed in `npd_game.py:54` and self correctly excluded (`npd_game.py:76-81`). HIGH.
- **Paper code — `single_run_3tfte_standalone.py:26-32`** (canonical, self excluded):
  ```python
  def nperson_payoff(my_move, num_other_cooperators, total_agents):
      if my_move == COOPERATE: return S + (R - S) * (num_other_cooperators / (total_agents - 1))
      else:                    return P + (T - P) * (num_other_cooperators / (total_agents - 1))
  ```
- **`cooperation_measurement.py:37-43`** correctly excludes self: `others_coop = num_cooperators - (1 - my_move)`.
- **⚠ Inconsistency — `final_simulation.py:10-14` + call site `:58`:** the same-named `nperson_payoff(my_move, num_other_cooperators, total_agents)` is **called with the *total* cooperator count** (`nperson_payoff(my_move, num_cooperators, len(agents))`) — self **not** excluded. For 3-of-3 cooperating this yields `0 + 3*(3/2) = 4.5 > R`, an impossible payoff. So `final_simulation.py` disagrees with the standalone/`cooperation_measurement` self-exclusion. Which "final" runner fed the paper matters. MED. **Flag for reproduction.**

### `npdl` global-bonus modification (non-standard)
- **`environment.py:192-198`** neighbourhood payoffs add a **quadratic global cooperation bonus** to cooperators only: `global_bonus = global_coop_rate**2 * 2` (max +2), on by default (`use_global_bonus=True`). This is **not** in the paper's linear model and inflates cooperator payoffs. Any `npdl` neighbourhood run left at defaults is **not** the paper's game. HIGH.

## 3. THE DEFECT (most important finding)

### Symptom
`test_results/pairwise_tft_cooperation_all_experiments_summary.csv` reports `3_TFT_mean = 0.0` for **all 20 rounds**; the neighbourhood twin reports `1.0` for all rounds. Documented in `CORRECTED_RESULTS_SUMMARY.md:5,31` as "0% cooperation for pairwise games… incorrect… a bug in how the experiment runner is tracking cooperation." HIGH (read the CSV).

### Root cause (verified in code) — a schema mismatch in the metric extractor
- The pairwise engine `PairwiseGame` (`npd_simulator/core/game/pairwise_game.py:114-133`) records history as **pair-indexed** dicts: keys `agent1_id, agent2_id, action1, action2, payoff1, payoff2, round, episode`. There is **no `agents` key and no `cooperation_rate` key** per record. `get_tournament_results()` (`:135-169`) returns that flat history.
- The runner `experiment_runner.run_pairwise_experiment` (`experiment_runner.py:195-262`) also plays each pair to completion (`:221-247`) — a full `rounds_per_pair` inner loop **per pair, sequentially**, not a synchronized global round.
- The metric extractor `research_experiments._extract_cooperation_by_round` (`research_experiments.py:238-266`):
  ```python
  for round_data in sorted_history:
      if "agents" in round_data:                       # ← pairwise records never have this
          ... tft_coops from agent_data["action"] ...
      else:
          cooperation_rates.append(round_data.get("cooperation_rate", 0))  # ← also absent → 0
  ```
  For pairwise history the `"agents"` branch is never taken and `"cooperation_rate"` is missing, so **`.get(..., 0)` returns 0 for every record**. `:263-264` then pads to `num_rounds` with more `0.0`. Result: identically-zero pairwise cooperation, round 1 included — proving it is a **reporting** bug (agents actually cooperate; the reader looks at the wrong schema and defaults to 0). HIGH.
- Contributing design flaw: even with a correct reader, "cooperation by round" is **ill-defined** for this pairwise runner because pairs are played sequentially to completion, so there is no common global round axis to align against the neighbourhood run. HIGH.

### What it contaminated
- `test_results/pairwise_tft_cooperation_all_experiments_summary.csv` and any per-run CSVs / figures produced by `visualize_research_results.py` off that path. The neighbourhood CSVs are *not* affected by this particular bug but are produced by a different code path with its own semantics.

### How/whether it was fixed
- **Not fixed in place.** `CORRECTED_RESULTS_SUMMARY.md` only *describes* the expected corrected picture; "Next Steps" list the fix as TODO. A parallel `static_style_runner.run_pairwise_experiment_enhanced` (`static_style_runner.py:132-200`) *does* build a proper synchronized round loop with `round_actions`/`EnhancedGameState`, which would report cooperation correctly — but that is a **different runner**, and it is unclear it regenerated the committed `test_results`. MED.
- **The real "fix" was to abandon this path** and generate paper figures with the standalone `final_experimentations`/`code_for_website` scripts, which track per-agent cooperation directly (`final_simulation.py:42` `agent_moves.count(0)/len(agent_moves)`; standalone tracks `1-my_move`). HIGH.

### Is the fix itself correct?
- The **standalone** cooperation tracking is correct (per-agent, per-round). HIGH.
- But it swaps in a **different confound** (pTFT vs per-opponent TFT — §4) and is **unseeded** (§5). So "corrected results" are trustworthy *as a different experiment*, not as a bug-for-bug fix of the `npd_simulator` numbers. MED.
- `static_style_runner`'s enhanced pairwise still keeps only the **most recent** action per agent per round for display (`static_style_runner.py:188-190`), which under-represents that an agent plays multiple simultaneous games — acceptable for a coop-rate proxy but lossy. LOW.

## 4. STRATEGY-SEMANTICS DIVERGENCE (co-equal to the defect for reuse)

Three incompatible "pairwise TFT" and two "neighbourhood TFT" definitions exist:

| Engine | Pairwise TFT | Neighbourhood TFT |
|---|---|---|
| `npdl/core` (`agents.py:71-117`, `environment.py:287-386`) | **One move/round for all opponents**; "defect if **ANY** specific opponent defected last round" (`agents.py:94`). A single AllD ⇒ all TFTs defect. | Threshold rule: cooperate iff coop-proportion of neighbours ≥ 0.5 (`agents.py:105-115`). Deterministic. |
| `npdl/core/true_pairwise.py` | **Per-opponent** decisions & memory (`OpponentSpecificMemory`). True reciprocity. Not used by `main.py`. | n/a |
| Paper standalone (`final_agents.py:33-45`, `single_run_3tfte_standalone.py:38-...`) | **Per-opponent** deterministic: `opponent_last_moves.get(opponent_id, COOPERATE)`. AllD isolated; TFT-TFT stays CC. | **pTFT**: `COOPERATE if random.random() < group_coop_ratio else DEFECT` (`final_agents.py:54-60`, standalone comment "This is the pTFT logic"). Stochastic. |

**Consequences for reuse:**
- The `npdl` `interaction_mode="pairwise"` **reproduces the *wrong* result** for the paper's central claim (pairwise collapses under one defector). It models *linked/aggregate* pairwise, not independent bilateral games. HIGH.
- The Hill-vs-Valley gap conflates **interaction structure** with **decision rule** (deterministic reciprocity vs probabilistic matching). A clean reimplementation must hold the strategy fixed across conditions, or explicitly treat pTFT as the object of study. HIGH.

## 5. RANDOMNESS / SEEDING

- **Per-agent RNG: none.** Everything draws from the global `random` module and `numpy.random`; shared global state. HIGH.
- **`npdl/main.py:334-335`** seeds per run: `random.seed(run_number); np.random.seed(run_number)` — reproducible within that runner. HIGH.
- **Paper/standalone code: no seeding at all.** `grep seed` over `single_run_3tfte_standalone.py`, `final_demo_full.py`, `final_simulation.py`, `cooperation_measurement.py` (v9) returns nothing (only `interactive_q_learning_trace.py` uses `np.random.random()` cosmetically). `run_multiple_simulations` (`cooperation_measurement.py:149-179`) just loops `num_runs` with fresh agents and **no seed** → **not exactly reproducible**; relies on averaging (default 10 runs × 10 000 rounds, `config.py`). HIGH.
- **Network construction** (`environment.py:80-158`) uses NetworkX generators seeded only by the global `random`/numpy state — so `npdl` network topology is reproducible only under `main.py`'s per-run seeding. MED.
- Reproducibility verdict: **npdl path = reproducible; paper path = distributionally reproducible only.** For a research-grade lab this is a gap to close (explicit seed plumbing, per-run seed logging).

## 6. TESTS — coverage vs appearance

21 test files, `pytest.ini` present. Genuine unit coverage of payoff tables, agent memory, network builders, Q-learning updates. But several tests **enshrine the divergent/aggregate semantics as "correct":**
- **`tests/test_neighborhood_vs_pairwise.py:67-70`**: asserts `final_moves_p[i] == "defect"` — i.e. it *requires* TFT to collapse to defection in `npdl` pairwise with a defector present. This **locks in the behavior that contradicts the paper's Reciprocity-Hill claim.** HIGH. Any refactor toward true pairwise will (correctly) break this test.
- **`tests/test_pairwise.py:80-128`**: uses a **0.99** cooperation threshold ("defect when proportion < 0.99") — testing the aggregate `opponent_coop_proportion` branch, a fixture-specific construction rather than a scientific behavior. Confirms plumbing, not the finding. MED.
- `test_pairwise.py:138` does assert the "defect if ANY specific opponent defected" branch — again validating the *linked* model.
- `test_true_pairwise.py` exercises the per-opponent engine separately (good), but that engine isn't the one wired into `main.py`.
- **No test** asserts the headline quantitative claims (3-TFT pairwise → ~100%; 2TFT+1AllD → ~66%; neighbourhood tragic valley). No regression test guards the cooperation-tracking extractor that caused the zero-bug. HIGH.
- `test_environment_fixed.py` vs `test_environment.py` coexist (a "fixed" duplicate) — smell of patched-around history.

**Net:** tests confirm the code does what the (divergent) engines were written to do; they do **not** validate the scientific claims and in places **cement the wrong pairwise semantics.**

## 8. REUSE VERDICT

### Safe to reuse (with light adaptation)
- **PD payoff primitives**: `npdl/core/utils.py` `linear_payoff_C/D`, `get_pairwise_payoffs`, and `npd_simulator/core/models/payoff_matrix.py` (has constraint validation T>R>P>S, 2R>T+S). Pin R/S/T/P and drop the duplicate `get_pairwise_payoffs`. HIGH.
- **`npd_simulator` modular skeleton** (game/model separation, config-driven runners) as an *architecture reference* — but not its metric layer.
- **`npdl/core/true_pairwise.py`** as the *correct* per-opponent reciprocity reference for the new pairwise engine. MED.
- Standalone `single_run_3tfte_standalone.py` as an executable spec of the **intended** pairwise (per-opponent TFT) and neighbourhood (pTFT) semantics.

### Must be reimplemented / must NOT be trusted as-is
- **Any `npdl` `interaction_mode="pairwise"` result** — it is linked-aggregate pairwise, not bilateral; reproduces the wrong claim. Reimplement as independent per-opponent games. HIGH.
- **The cooperation-tracking / reporting layer** (`research_experiments._extract_cooperation_by_round`) — schema-fragile, silently defaults to 0. Replace with a typed, fail-loud metrics layer with a single canonical round schema. HIGH.
- **The `environment.py` quadratic global-bonus** default — remove or make it an explicit, off-by-default mechanism; it silently changes the game. HIGH.
- **Committed `test_results/pairwise_*` CSVs** — contaminated; discard, do not use as fixtures or baselines. HIGH.
- **Strategy definitions across conditions** — do not port "TFT" blindly; the neighbourhood variant is pTFT. Decide explicitly whether the object of study is TFT or pTFT and hold it fixed (or vary it deliberately). HIGH.
- **Reproducibility** — add explicit per-run seed plumbing (global `random` + `numpy` + any per-agent RNG) and record seeds in manifests; the paper path had none. HIGH.

### Provenance / OUTPUTS traceability (§6 deliverable)
- Artifacts: `test_results/*.csv` (npd_simulator research path, **contaminated pairwise**), `demo_results/batch_runs/batch_2025*` (timestamped, some empty per `visualization_fix_summary.md`), `parameter_sweep_results/`, `pictures/*.png` (historical, **no embedded commit/provenance**), `Paper Resources/`, and the committed paper PDF (`ee932701`).
- **Traceability is weak:** figures/CSVs carry timestamps but **no code version / git hash / seed / config hash**. Multiple engines could have produced any given figure. Any reproduction must be treated as *re-derivation from claims*, not *bit-repro of artifacts*. HIGH.
